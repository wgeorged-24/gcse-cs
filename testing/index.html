‚úàÔ∏è Why Do We Need Test Plans?

Imagine you've written a program that calculates exam grades. You run it once with your own marks, it works, so you assume it's finished. But what about:

  • What if someone enters a negative mark?
  • What if they enter 100 exactly (the boundary)?
  • What if they type "fifty" instead of 50?
  • What if the mark is 101?

Without a test plan, you might miss these cases entirely.

‚úàÔ∏è Pre-Flight Checklist Analogy

Pilots don't just start the engine and hope the plane works. Before every flight, they work through a pre-flight checklist ‚Äî a systematic list of everything that must be checked.

A test plan is your program's pre-flight checklist. It ensures you check everything systematically, not just the obvious things.

Key Term Test Plan

A structured document that lists all the tests to be performed on a program, including what data to use, what results to expect, and whether each test passed or failed.

‚ö†Ô∏è Common Mistake

Students often test only with "normal" data that they expect to work. A good test plan deliberately includes data that should cause errors ‚Äî to check the program handles them correctly.

üîÑ Quick Recap: Types of Test Data

Before we build test plans, let's quickly recap the four types of test data you'll use:

Normal Data

Typical, expected inputs that should be accepted.

Age (0-120): 25

Boundary Data

Values at the edges of valid ranges.

Age (0-120): 0, 120

Invalid Data

Outside range but correct data type.

Age (0-120): -5, 150

Erroneous Data

Completely wrong data type.

Age: "twenty"
‚úàÔ∏è Pre-Flight Checklist Analogy

Think of these as different types of checks: normal (engine sounds right), boundary (fuel exactly at minimum), invalid (fuel gauge impossible value), erroneous (water in fuel tank).

ü§î Why Test Invalid and Erroneous Data?

You're not testing whether wrong data works ‚Äî you're testing whether your program handles wrong data correctly (showing an error message rather than crashing).

üìã Structure of a Test Plan

A test plan is usually presented as a table with specific columns:

Test No. Description Test Data Data Type Expected Result Actual Result Pass/Fail
1 Test typical valid age 25 Normal Accepted, processed Accepted, processed Pass
2 Test lower boundary 0 Boundary Accepted Accepted Pass
3 Test negative age -5 Invalid Error message Program crashed Fail
Exam Tip Expected vs Actual

Expected Result: What should happen (written before testing).
Actual Result: What did happen (filled in after testing).
If they match = Pass. If not = Fail.

‚úàÔ∏è Pre-Flight Checklist Analogy

Each row is like one item on the checklist. "Expected Result" is what should happen (fuel gauge shows full). "Actual Result" is what you observe. If they match, tick "Pass"!

‚úì Quick Check

üìù Notes for This Section

Test Plan Columns

  • Test No.: Unique identifier
  • Description: What is being tested
  • Test Data: The actual value used
  • Data Type: Normal/Boundary/Invalid/Erroneous
  • Expected Result: What should happen (before)
  • Actual Result: What did happen (after)
  • Pass/Fail: Do expected and actual match?

‚úçÔ∏è Writing Good Test Descriptions

The description should explain what you're testing and why:

‚ùå Poor Descriptions

  • "Test the program"
  • "Enter 50"
  • "Check it works"

‚úì Good Descriptions

  • "Test with typical valid percentage in middle of range"
  • "Test upper boundary of valid range (100)"
  • "Test that text input is rejected with error message"
Formula Description Template

"Test [what/where] with [type of data] to check [expected behaviour]"

Example: "Test age input with boundary value (0) to check minimum is accepted"

‚ö†Ô∏è Exam Warning

Vague descriptions like "test with normal data" lose marks. Be specific about what and why.

‚úì Quick Check

üìù Notes for This Section

Good Test Descriptions

  • Be specific about what you're testing
  • Explain why you chose that data
  • Mention the type of test
  • Template: "Test [what] with [data type] to check [behaviour]"

üéØ Determining Expected Results

The expected result describes what should happen:

Data Type Expected Result (typically)
Normal Data accepted, correct output produced
Boundary Data accepted (if in range), correct output for edge case
Invalid Data rejected, error message displayed
Erroneous Data rejected, error asking for correct type
ü§î Be Specific!

Don't write "Error message" ‚Äî write what it should say: "Error: Enter a number between 0 and 100"

Don't write "Accepted" ‚Äî write what happens: "Grade 'B' displayed"

‚úàÔ∏è Pre-Flight Checklist Analogy

Pilots know exactly what "normal" looks like. "Fuel gauge" isn't just "check" ‚Äî it's "needle in green zone, at least 75%". Your expected results should be equally specific.

‚úì Quick Check

üìù Notes for This Section

Expected Results

  • Normal/Boundary: Data accepted + specific output
  • Invalid/Erroneous: Data rejected + error message
  • Be specific about what should happen
  • Written before testing

üìã Scenario

A program calculates grades from percentage marks (0-100):

70-100 = A 60-69 = B 50-59 = C 40-49 = D 0-39 = F
No. Description Data Type Expected Actual P/F
1 Test typical B grade mark 65 Normal Grade "B" Grade "B" Pass
2 Test lower boundary (0) 0 Boundary Grade "F" Grade "F" Pass
3 Test upper boundary (100) 100 Boundary Grade "A" Grade "A" Pass
4 Test grade boundary (69/70) 70 Boundary Grade "A" Grade "A" Pass
5 Test above maximum (101) 101 Invalid Error message Error shown Pass
6 Test negative value -5 Invalid Error message Crash Fail
7 Test text input "fifty" Erroneous Error message Error shown Pass
‚ö†Ô∏è Notice Test 6

Test 6 failed ‚Äî the program crashed instead of showing an error. The test plan found a bug that needs fixing!

Test Plan Coverage

üìã Summary

The 7 Columns

  1. Test Number ‚Äî Unique identifier
  2. Description ‚Äî What and why
  3. Test Data ‚Äî The value
  4. Data Type ‚Äî Normal/Boundary/Invalid/Erroneous

  • Erroneous wrong data type
